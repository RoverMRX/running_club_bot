## Архитектура проекта: IT Running Bot

### 1\. Модуль Core (Ядро)

Этот модуль отвечает за инициализацию бота, управление состояниями (FSM) и ролевую модель.

### 2\. Модуль Database (Модели данных)

Для реализации твоих идей потребуется гибкая структура в SQLAlchemy.

### 3\. Модуль Challenge Manager (Логика целей)

| Тип челленджа | Как работает | Как проверяется |

### 5\. Модуль Integration (Strava API)

- **OAuth Flow:** Веб-сервер (на `Aiohttp` или `FastAPI`), который ловит callback от Strava.
- **Webhook Listener:** Чтобы бот сам писал в чат: «О, @user только что пробежал 10км!», как только тренировка синхронизировалась.

### 6\. Модуль Reporting (Дайджесты)

## 3\. Описание модулей (Логика работы)

### config.py

Использует `pydantic-settings`. Его задача — убедиться, что все ключи на месте. Если ты забудешь прописать токен в `.env`, бот выдаст ошибку сразу при запуске, а не в процессе работы.

### database.py

Здесь описан "скелет" данных. Мы используем асинхронный движок `SQLAlchemy`.

- **Таблица Users:** хранит XP, роль и статистику.
- **Таблица Challenges:** хранит активные цели, дату следующей проверки и текст штрафа.
- **Таблица Reports:** логирует каждый присланный отчет для истории.

### handlers.py

Основной блок кода.

1. **Command Handlers:** Реагируют на `/start`, `/challenge`, `/stats`.
2. **FSM Handlers:** Ведут пользователя по шагам создания цели (Тип -> Значение -> Штраф -> Дата).
3. **Callback Handlers:** Обрабатывают нажатия на кнопки «Засчитать» или «Отклонить» отчет.

### bot.py

Склеивает всё воедино.

- Подключает роутеры из `handlers.py`.
- Запускает базу данных (`init_db`).
- Настраивает прокси для обхода блокировок.

**Gemini**

Если объяснять просто, **.env** — это текстовый файл, который служит «сейфом» или «панелью управления» для твоего приложения. В нем хранятся настройки и секретные данные, которые не должны быть жестко прописаны в самом программном коде.

Вот основные моменты, которые объясняют его суть:

### 1\. Формат хранения

Данные в этом файле хранятся в виде пар **КЛЮЧ=ЗНАЧЕНИЕ**. Например:

```
BOT_TOKEN=74839201:AAF-ExampleToken
DB_URL=sqlite:///it_run.db
ADMIN_ID=12345678
DEBUG=True
```

### 2\. Зачем это нужно?

- **Безопасность (главное!):** Ты никогда не должен выкладывать свои пароли, API-ключи или токены в публичный доступ (например, на GitHub). Код ты выкладываешь, а файл `.env` — **нет** (его добавляют в исключения через файл `.gitignore`). Это предотвращает кражу твоих ключей.
- **Гибкость:** Тебе не нужно менять код, чтобы изменить настройки. Достаточно поправить значение в одном текстовом файле.
- **Разделение сред:** У тебя может быть один `.env` для тестов на твоем компьютере (где база данных лежит локально), и совсем другой `.env` на сервере (где база данных защищена паролем). При этом сам код бота остается неизменным.

### 3\. Как это работает в коде?

Когда твоя программа запускается, она обращается к этому файлу, считывает его и загружает данные в **переменные окружения** (Environment Variables).

В Python для этого чаще всего используют библиотеки:

- `python-dotenv` (классика).
- `pydantic-settings` (более современный и надежный способ, который мы выбрали для нашего проекта).

### 4\. Почему файл начинается с точки?

В операционных системах на базе Linux (и macOS) файлы, начинающиеся с точки, считаются **скрытыми**. Это сделано специально, чтобы такие системные или конфигурационные файлы не мешались в общей папке и их было сложнее случайно удалить или изменить.

### Шаг 4: База данных database.py

Здесь мы описываем таблицу челленджей. Файл базы `it_run.db` создастся сам при первом запуске.

```
from datetime import datetime
from sqlalchemy import BigInteger, Column, Integer, String, Float, DateTime, Boolean, ForeignKey
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession, async_sessionmaker
from sqlalchemy.orm import declarative_base

from config import config

engine = create_async_engine(config.DB_URL)
async_session = async_sessionmaker(engine, expire_on_commit=False)
Base = declarative_base()

class Challenge(Base):
    __tablename__ = 'challenges'
    id = Column(Integer, primary_key=True)
    user_id = Column(BigInteger)
    user_name = Column(String)
    type = Column(String)       # 'Разы' или 'Км'
    goal_value = Column(Float)
    penalty = Column(String)
    start_date = Column(DateTime, default=datetime.utcnow)
    is_active = Column(Boolean, default=True)

async def init_db():
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
```

### Шаг 5: Мозги бота handlers.py

Тут живет логика анкеты. Обрати внимание: в конце мы реально сохраняем данные в базу.

```
from aiogram import Router, types, F
from aiogram.filters import Command
from aiogram.fsm.context import FSMContext
from aiogram.fsm.state import StatesGroup, State
from database import async_session, Challenge

router = Router()

class ChallengeForm(StatesGroup):
    choosing_type = State()
    input_value = State()
    input_penalty = State()

@router.message(Command("challenge"))
async def cmd_challenge(message: types.Message, state: FSMContext):
    await state.set_state(ChallengeForm.choosing_type)
    kb = types.ReplyKeyboardMarkup(
        keyboard=[[types.KeyboardButton(text="Разы в неделю"), types.KeyboardButton(text="Дистанция (км)")]],
        resize_keyboard=True
    )
    await message.answer("Выбирай тип челленджа:", reply_markup=kb)

@router.message(ChallengeForm.choosing_type)
async def process_type(message: types.Message, state: FSMContext):
    await state.update_data(type=message.text)
    await state.set_state(ChallengeForm.input_value)
    await message.answer("Введи цель (число):", reply_markup=types.ReplyKeyboardRemove())

@router.message(ChallengeForm.input_value)
async def process_value(message: types.Message, state: FSMContext):
    try:
        val = float(message.text.replace(',', '.'))
        await state.update_data(value=val)
        await state.set_state(ChallengeForm.input_penalty)
        await message.answer("Что проставишь в случае провала? (например: Пицца)")
    except ValueError:
        await message.answer("Введи число (например: 3 или 20.5)")

@router.message(ChallengeForm.input_penalty)
async def process_penalty(message: types.Message, state: FSMContext):
    data = await state.get_data()
    
    # Сохраняем в БД
    async with async_session() as session:
        new_ch = Challenge(
            user_id=message.from_user.id,
            user_name=message.from_user.username or message.from_user.first_name,
            type=data['type'],
            goal_value=data['value'],
            penalty=message.text
        )
        session.add(new_ch)
        await session.commit()

    await state.clear()
    await message.answer(f"✅ Челлендж записан!\nЦель: {data['value']} {data['type']}\nШтраф: {message.text}")
```

### Шаг 6: Файл запуска bot.py

Склеиваем всё.

```
import asyncio
import logging
from aiogram import Bot, Dispatcher
from config import config
from database import init_db
from handlers import router

async def main():
    logging.basicConfig(level=logging.INFO)
    await init_db()
    
    bot = Bot(token=config.BOT_TOKEN, parse_mode="HTML")
    dp = Dispatcher()
    dp.include_router(router)

    print("🚀 Бот запущен локально!")
    await dp.start_polling(bot)

if __name__ == "__main__":
    asyncio.run(main())
```

import logging
from aiogram import Bot, Dispatcher
from aiogram.client.session.aiohttp import AiohttpSession
from aiogram.client.default import DefaultBotProperties
from aiohttp_socks import ProxyConnector  # Добавляем этот импорт

from database import init_db
from handlers import router

async def main():

    await init_db()
    
    session = None
    if config.PROXY_URL:
        # Проверяем, если это SOCKS прокси
        if config.PROXY_URL.startswith("socks"):
            connector = ProxyConnector.from_url(config.PROXY_URL)
            session = AiohttpSession(connector=connector)
        else:
            # Для обычных HTTP прокси
            session = AiohttpSession(proxy=config.PROXY_URL)

    await dp.start_polling(bot)

if __name__ == "__main__":

## 1\. Логика P2P подтверждения (Голосование)

## 7\. Технический P2P протокол (Consensus Logic)

Чтобы избежать фрода, мы внедрим **Double-Check**:

1. Голосовать могут только те, у кого у самих есть активный челлендж (защита от "левых" аккаунтов).
2. Бот записывает `message_id` отчета и `user_id` проголосовавших в таблицу `Votes`.
3. При каждом нажатии делается SQL запрос: `SELECT COUNT(*) FROM Votes WHERE report_id = ...`.

## 3\. ЦИКЛ ЖИЗНИ ЧЕЛЛЕНДЖА

Челлендж — это не просто запись, это процесс.

### Шаг 1: Заявление (The Commitment)

Юзер вызывает команду `/challenge`. Бот через анкету собирает данные:

- **Тип:** Дистанция (км) или Количество (раз в неделю).
- **Длительность:** По умолчанию — недельный спринт (до ближайшего воскресенья, 23:59).
- **Значение:** Например, 20 км.
- **Наказание:** «Куплю 3 кг бананов на следующую общую пробежку».

### Шаг 2: Активность (Reporting)

Юзер бегает и присылает в чат фото (скриншот трекера).

- Бот подтягивает текущий прогресс юзера: *«Пробежал 5 км из 20 км»*.

### Шаг 3: Модерация (P2P Verification)

Под каждым отчетом бот создает панель: `[✅ Засчитать (0/3)]`.

- Участники жмут кнопку. При наборе 3-х голосов прогресс засчитывается официально.
- Админ может нажать кнопку, и отчет засчитается мгновенно (1/1).

### Шаг 4: Итоги (The Deadline)

В воскресенье в 23:59 бот подводит итоги:

- **Success:** Юзер получает XP (Опыт) и растет в уровне.
- **Fail:** Юзер попадает в «Список штрафников». Ему запрещено брать новый челлендж, пока админ не подтвердит выполнение штрафа.
